A change in thinking

From task and response
to systems and design

Task-and-Response Thinking

Translate this.→Summarise this.→Tell me about this.

You work forward through requests and responses. Each answer may suggest what to ask next.

Systems-and-Design Thinking

purposepeopleinformationpartssuccess

You begin with the whole. You work backward to understand and design its parts, then build forward in planned stages.

The shift is not from small tasks to large tasks. It is from discovering the work through a chain of requests to designing the whole before decomposing it into tasks.

A useful term

What is vibe coding?

Vibe coding is a term used to describe creating software by explaining what you want in ordinary language.

The AI produces and changes the code while you concentrate mainly on what the finished tool should do. You try what it creates, then tell it what to change.

That is what we are doing today. We are vibe coding.

Our approach

Vibe coding with a map

Before we build, we step back and think about the whole system. We work backward to understand and design its parts, then build forward in planned stages.

WHOLEDESIGNPARTSBUILD

This preparation does not remove experimentation or creativity. It makes the process more intentional and reduces the time spent stabbing in the dark.

The method

Design backward.
Build forward.

These ideas are not entirely new. Systems thinking, product design, and software engineering have developed related ways of planning complex work for decades. Agentic tools are making those practices useful to a much wider group of people.

01

Context engineering

Understand the whole

Surface and organise the purpose, people, needs, constraints, source material, and desired result before anything is built.

You do not have to know all the answers. You do need a process that helps you discover the right questions.

Try this with an agent

Interview me one question at a time about the tool I want to make. Help me uncover its purpose, users, needs, constraints, source material, and desired result before we build anything.

Matt Pocock skill $grill-with-docs ↗ It interviews you and records the decisions as you go.
02

Build Blueprint

Write the blueprint

Create a short plain-language document describing who the tool serves, what it must do, and what must be true for it to count as complete.

A blueprint gives the agent a stable view of the whole before implementation begins.

Try this with an agent

Turn everything we have agreed into a clear blueprint. Describe who the tool is for, what it must do, what is outside the plan, and how we will know it is complete.

Matt Pocock skill $to-prd ↗ It turns the conversation into a clear build blueprint.
03

Task decomposition

Break it into buildable parts

Turn the blueprint into focused stages that each produce something testable.

Small stages create useful feedback. They also make it easier to identify where something has gone wrong.

Try this with an agent

Break this blueprint into small, ordered pieces of work. Each piece should create something useful that can be checked before the next piece begins.

Matt Pocock skill $to-issues ↗ It turns the blueprint into buildable pieces of work.
04

Implementation

Build and check each stage

Build the planned parts one at a time, check that each works, then combine them into the intended whole.

The plan guides the work. Testing tells you what is actually true.

Try this with an agent

Build the planned pieces one at a time. Check each piece against the blueprint, test that it works, and stop if an important decision is missing.

Matt Pocock skill $implement ↗ It carries out the planned work and checks the result.

A practical starting move

Ask the agent to interview you before it builds.

Grill me about every aspect of this idea. Ask one question at a time and give me your recommended answer. Do not start building until we agree on what should be created.

This presentation website was planned using that same process before any code was written.

Before you upload anything

If the material is confidential, pause.

Do not upload it until you understand the account, settings, permissions, and connected services involved. When unsure, use public, synthetic, or expressly authorised sample material.

A sandbox limits what software can touch. It does not automatically keep your information on your computer or make it confidential.

This is practical risk-management guidance, not legal advice. Your client agreement, professional obligations, and applicable law still govern the work.

Where does the work run?

Remote Cowork is now the documented default. Its agent loop and code run in a temporary isolated environment on Anthropic infrastructure. Older local Cowork deployments run the agent loop and connected-folder operations locally while code runs in a dedicated Linux virtual machine.

Claude Code can run against a local project, in Anthropic’s cloud, over SSH, or through Windows Subsystem for Linux. Local Code is permission-gated but is not automatically a virtual machine.

Cowork architecture ↗ Claude Code sandboxing ↗
Where can the data go?

Remote Cowork processes opened local files on Anthropic servers. Local Claude Code commands run locally, but prompts, relevant file content, outputs, and code context are sent for model processing.

On consumer accounts, conversations can be used for model improvement when Model Improvement is enabled. Incognito chats are not used for model improvement. Commercial chats and coding sessions are not used for training by default.
